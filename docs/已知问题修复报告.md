# 截图工具已知问题排查与修复报告

> 日期：2026-07-12 ｜ 范围：截图捕获管线 / 剪贴板 / 滚动截图 / 透明通道
> 参考高星项目：Flameshot (flameshot-org/flameshot)、ShareX、JamTools

## 一、排查方法

1. **检索同类项目已知缺陷**：查阅 Flameshot issue #564 / #4171（分数缩放下选区错位）、ShareX 剪贴板透明处理、JamTools 滚动拼接实现，确认截图工具的高频缺陷类别。
2. **静态代码审计**：逐文件审查 `engines/capture/*`、`controllers/screenshot_controller.py`、`controllers/editor_controller.py`、`utils/image_utils.py`，定位坐标换算、多屏、越界、透明通道等隐患。
3. **单元测试验证**：为每个问题编写可复现的回归测试（纯逻辑 + offscreen 往返），确保修复可验证、不回退。

## 二、发现的问题与修复

### 问题 1：HiDPI / 分数缩放下区域/窗口截图坐标错位（高）
- **现象**：当 Windows 缩放设为 125% / 150% 时，截取的区域位置偏移、尺寸错误（只截到屏幕一部分）。
- **根因**：`QML`/UI 坐标为**逻辑像素**，而 `QScreen.grabWindow()` 按**设备像素**抓取；原代码直接把逻辑坐标传入 `grabWindow`，未乘 `devicePixelRatio`。`ScreenshotData.dpi_scale` 字段一直默认 `1.0` 且从未接线，印证该缺陷。
- **参考**：Flameshot issue #564 / #4171 —— 分数缩放下必须 `选区 × devicePixelRatio` 才正确。
- **修复**：新增纯函数 `device_rect(logical_rect, dpr)`（`utils/capture_geometry.py`）。在 `RegionCapture.captureRect`、`ScreenshotController._doRegionCapture`、`WindowCapture.captureWindow` 中，将逻辑坐标（或相对屏幕偏移）乘 `dpr` 后抓取。
- **验证**：`test_dpr_1_5_scales` 等 4 个用例断言 (100,100,200,100)@1.5 → (150,150,300,150)。

### 问题 2：多显示器下区域截图只能选主屏（高）
- **现象**：副屏上无法框选截图；覆盖层只铺满主屏。
- **根因**：`_showRegionOverlay` 仅用 `primaryScreen().geometry()`，覆盖层与抓取都局限主屏；`grabWindow(0)` 也只用主屏。
- **参考**：Flameshot 为每块屏幕生成捕获；ShareX 按坐标落在哪块屏幕决定抓取来源。
- **修复**：
  - `virtual_bounding_geometry(screens)` 计算所有屏幕逻辑几何并集，覆盖层铺满虚拟桌面并合成多屏背景。
  - `screen_index_for_point(point, screens)` 按选区左上角选对屏幕，`_doRegionCapture` 用该屏幕 `grabWindow` 并换算相对坐标。
- **验证**：`test_virtual_bounding_geometry` / `test_screen_index_*` 断言并集几何与选屏正确（单屏时行为与原来一致，零回退）。

### 问题 3：选区越界导致 `grabWindow` 返回空图（中）
- **现象**：选区略微超出屏幕（DPI 取整/边界）时，抓取返回空 pixmap，截图被静默取消。
- **根因**：未对抓取矩形做边界裁剪。
- **修复**：`clamp_rect_to_geometry(rect, geom)` 将设备像素矩形裁剪进屏幕设备几何（宽高至少 1）。接入区域与窗口抓取。
- **验证**：`test_overflow_right_bottom_clamped` / `test_negative_origin_clamped` / `test_min_size_one`。

### 问题 4：滚动截图在 ORB 拼接中途失败时只输出首帧（中/高）
- **现象**：长页面滚动截图，若某次特征匹配失败，最终产物只剩第一帧，后续内容全部丢失。
- **根因**：`_finishCapture` 中 `result = frames[-1] if len==1 else frames[0]` —— 拼接失败时 `frames` 会 `append` 增长，`len>1` 却取 `frames[0]`（最早一帧），丢弃累积结果。
- **参考**：JamTools 滚动拼接回退为按序纵向拼接原始帧。
- **修复**：新增 `_vstack_frames(frames)`，多帧时按捕获顺序纵向拼接（统一最大宽度、透明背景）；正常流程 `frames` 恒为 1、`frames[-1]` 即完整图，行为不变。
- **验证**：`test_vstack_*` / `test_finish_prefers_stacked_when_multi` 断言两帧拼接高度为两者之和而非首帧高度。

### 问题 5：复制到剪贴板丢失透明通道（中）
- **现象**：带圆角/透明的截图复制到剪贴板后，在部分应用粘贴成黑底。
- **根因**：`copyToClipboard` 仅 `clipboard.setPixmap()`，Windows 上只写 `CF_BITMAP`（24 位），丢弃 alpha。
- **参考**：ShareX 额外写入 `image/png` MIME，支持透明度的应用可取得带 alpha 的图像。
- **修复**：构造 `QMimeData`，同时 `setImageData` 与 `setData("image/png", png_bytes)`，再 `setMimeData`。
- **验证**：`test_clipboard_keeps_png_alpha` 经 offscreen 剪贴板往返，透明像素 alpha 仍为 0。

### 问题 6：`pixmapToPil` 始终丢弃 alpha 通道（高，根因级）
- **现象**：窗口截图（圆角）、透明叠加区域等，经转换后变为不透明黑底——所有捕获（全屏/区域/窗口）都受影响，也是问题 5 的上游。
- **根因**：`utils/image_utils.py` 第 58 行 `arr[..., :3]` 显式丢弃 alpha，强制返回 RGB。
- **修复**：返回 `Image.fromarray(arr.copy(), "RGBA")` 保留透明通道；同步修正 `scrolling.py` 中 `cv2.cvtColor(..., RGB2BGR)` 对 4 通道报错处（先 `.convert("RGB")`）。
- **验证**：`test_pixmap_to_pil_keeps_alpha` 断言模式为 RGBA 且透明像素 alpha=0；全量回归 121 项通过，证明无下游回退。

### 问题 7：Windows 控制台 GBK 编码导致日志 `UnicodeEncodeError` 运行时异常（低/中）
- **现象**：程序启动注册全局热键时，日志输出含 emoji（✅/❌/⚠️）触发 `UnicodeEncodeError: 'gbk' codec can't encode character '\u2705'`，日志流抛出异常（被 logging 捕获不崩溃，但属运行时异常，违反"无运行时异常"验收标准）。
- **根因**：Windows 控制台默认编码为 GBK，`logger` 写入含非 GBK 字符（emoji）的内容时会抛编码异常。
- **修复**：
  - 剔除全部日志调用中的 emoji（`platforms/native_hotkey.py` 的 ✅/❌、`main.py` 的 ⚠️），改为纯文本。
  - 在 `main.py` 顶部加 UTF-8 安全网：启动即 `sys.stdout.reconfigure(encoding="utf-8", errors="replace")` / `sys.stderr` 同理，对任何非 ASCII 日志内容做全局兜底，杜绝同类异常复发。
- **验证**：重新打包后 offscreen 启动日志中不再出现 `UnicodeEncodeError` Traceback；启动流程连贯无异常。

## 三、测试结果

- 新增 `tests/test_capture_issues.py`：**17 项**专项测试覆盖上述 6 类问题（问题 7 为日志/编码层，由启动验证覆盖）。
- 全量回归：**121 项全部通过**（原 104 + 新增 17），耗时 ~0.73s，零功能回退。

## 五、遗留说明（非阻断）

- 多屏覆盖层采用单窗口铺满虚拟桌面方案；在异 DPI 多屏下 Qt 单 HWND 跨屏 DPI 处理有已知限制，选区坐标已正确换算，覆盖层视觉在极端异比例下可能需进一步按屏拆分窗口（Flameshot 做法），留待后续优化。
- ORB 拼接失败回退为简单纵向拼接，重叠区可能重复计数，属降级行为；优先保证不丢帧。
