## 二、主要结构问题（含证据）

### 🔴 P1 — `core/` 与 `engines/` 的 "engine" 命名冲突、职责模糊
- `core/annotation_engine.py`（编排/注册表，动态字符串导入 `engines.annotation.*`）与 `engines/annotation/`（真实工具实现）都叫 "engine"。
- `core/screenshot_engine.py`（编排）与 `engines/capture/`（实现）同理。
- 读者无法一眼区分「谁是后端、谁是应用层」。Flameshot/Umi-OCR 把后端实现（tools/capture）与 app 编排（core）严格分目录，不用同名。

### 🔴 P1 — 导出模块死代码 + 逻辑重复
- `core/export_manager.py`：**全仓零引用**（已确认）。
- `engines/export/{base,png,jpg,pdf}_exporter.py`：**全仓零引用**（仅理论上经已死的 export_manager 可达）。
- `controllers/editor_controller.py:343-378` `saveToFile` **内联重复**实现导出（JPEG/BMP/PDF/PNG），与 engines/export 完全脱节。
- 结论：导出逻辑分散、一套死代码无人用。应合并为单一导出路径。

### 🟠 P2 — 配置存在于两处，易漂移
- 根 `config/` 与 `data/config/` 内容相同；`core/config_manager.py:43` 运行时读 `data/config/`，根 `config/` 是未被读取的重复种子。两者不同步会导致「改了没生效」。

### 🟠 P2 — 平台专属代码未隔离
- `utils/win32_utils.py`、`utils/native_hotkey.py` 是 Windows 专属；与跨平台 `image_utils/logger/app_dir/hotkey_manager` 混在一起。Umi-OCR 用独立 `platform/` 包隔离，便于未来多平台与可测试性。

### 🟡 P3 — 基类位置与命名细节
- `engines/base_tool.py`（标注工具基类）放在 `engines/` 顶层，而所有子类在 `engines/annotation/`。Flameshot 把基类与工具同目录。
- `models/tool.py` 实为「工具默认配置」(`ToolConfig`+`DEFAULT_TOOL_CONFIGS`)，名为 `tool` 易与 `ToolType` 混淆；更宜叫 `tool_config.py`。
- `views/` 无 `__init__.py`：QML 非 Python 包，**属正常**，非问题。

## 三、参考高星项目的最佳实践

| 项目 | 关键组织方式 | 本项目可借鉴 |
|------|------------|------------|
| **Flameshot** | `src/` 根；`src/core`、`src/tools`(每工具一文件)、`src/capture`、`src/widgets`、`src/gui`、`src/utils`、`src/config` | 后端(tools/capture)与 UI 分离；工具扁平单文件 |
| **ShareX** | 每关注点独立程序集：Core / ScreenCaptureLib / ImageEffectsLib / HistoryLib | 关注点分离为独立包 |
| **Umi-OCR** | `py_src/` 源根；`platform/`(win32/linux/mac) 隔离系统代码；`plugins/`、`core/` | 平台代码独立 `platform/` 包 |
| **JamTools** | `Capture` / `Annotation` / `Tools` 模块清晰分层 | 截图/标注/工具分层 |

**共性原则**：① 源码集中于源根；② 后端实现与 UI 严格分离；③ 平台代码独立包；④ 单一配置源；⑤ 死代码清零；⑥ 工具/后端按类型扁平分组。

## 四、优化方案

### 方案 A（推荐）：原地保守重构（保持根级包，最低风险）

1. **消除 engine 命名冲突**：`core/annotation_engine.py` → `core/annotation_service.py`；`core/screenshot_engine.py` → `core/capture_service.py`。（仅改 2 文件名 + 更新 import 站点，爆炸半径小）
2. **导出合并（单一路径）**：删除死代码 `core/export_manager.py`；改造 `editor_controller.saveToFile` 委托给 `engines/export/*`（复活 Png/Jpg/PdfExporter）。消除重复，输出需与原 `rendered.save(...)` 一致（用现有 test_rendering 校验）。
3. **配置单一源**：保留 `config/` 为唯一种子；`data/config/` 改为构建/首次运行时由 `config/` 拷贝生成，不再手维护副本；`config_manager` 仍读运行时 `data/config/`。
4. **平台隔离**：`utils/win32_utils.py`+`utils/native_hotkey.py` → `platform/win32_utils.py`+`platform/native_hotkey.py`；更新全部 import。
5. **基类归位**：`engines/base_tool.py` → `engines/annotation/base.py`（与 12 个工具同目录）；更新 `engines/annotation/*` 与 `core/annotation_service` 的 import。
6. **命名澄清**：`models/tool.py` → `models/tool_config.py`；更新引用。
7. 更新 `ScreenshotTool.spec` 的 `hiddenimports`（如新增 `platform` 包、移除已删模块引用）；`run_tests.py` 无需改。

**目标结构（方案 A）**：
```
core/      → annotation_service, capture_service, history_manager, undo_manager,
             event_bus, config_manager        # 应用编排层（无 "engine" 歧义）
engines/   → annotation/(base + 12 tools), capture/(4), export/(复用)
models/    → annotation, screenshot, tool_config
platform/  → win32_utils, native_hotkey        # 从 utils 拆出
utils/     → image_utils, logger, app_dir, hotkey_manager
views/ themes/ tests/ config/ data/  (不变)
```

### 方案 B（可选，更彻底）：完整 `src/` 迁移
- 所有 Python 包移入 `src/`（`src/core`、`src/engines`…），仿 Umi-OCR `py_src/` / Flameshot `src/`。
- 优点：源/资源边界最清晰，最贴近高星项目。
- 代价：**高**：需改 `ScreenshotTool.spec` 的 pathex/hiddenimports、所有 import 根、可能触发 PyInstaller 收集差异；回归面大。
- 建议：若您追求「与高星项目完全一致」，选 B；若以「安全、功能不变、快速交付」优先，选 A。
