## P0 — 崩溃级

### #1 `scrolling.py:119` — cv2 模块名被变量遮蔽（`NameError` 崩溃）

**文件**: `engines/capture/scrolling.py`, `_stitchFrames()` 方法

```python
# ❌ 修复前 — 第119行 cv2 被 numpy 数组覆盖
cv1 = cv2.cvtColor(np.array(img1), cv2.COLOR_RGB2BGR)  # cv1 = numpy 数组
cv2 = cv2.cvtColor(np.array(img2), cv2.COLOR_RGB2BGR)  # cv2 不再是 OpenCV 模块！
# ... 后续所有 cv2.xxx() 调用都会抛出 AttributeError
```

**技术指导**: 这是一个经典的**变量遮蔽（variable shadowing）**bug。Python 中赋值会创建局部变量——`cv2 = ...` 让局部命名空间的 `cv2` 不再指向导入的模块。任何后续 `cv2.ORB_create()` 都会抛出 `AttributeError: 'ndarray' object has no attribute 'ORB_create'`。

**✅ 修复**: 重命名为 `cv1_arr` / `cv2_arr`，不与模块名冲突。

**防范**:  
- 避免用模块名作为变量名（`numpy = ...`, `os = ...`, `cv2 = ...`）  
- 启用 IDE 的"变量名遮蔽内置/导入"警告  
- `flake8` 规则 F402 可检测此类问题

## P1 — 高危

### #2 `print()` 代替 `logger`（3个文件，共6处）

**影响**: PyInstaller 打包后 `print()` 输出到 `stdout` 缓冲区，EXE 无控制台时日志不可见，排查问题困难。

**文件**:  
- `themes/theme_manager.py:51,57,59`  
- `core/config_manager.py:59,76,147`  

```python
# ❌ 修复前
print(f"[ThemeManager] 加载主题失败 {theme_file}: {e}")

# ✅ 修复后
logger.warning("[ThemeManager] 加载主题失败 %s: %s", theme_file, e)
```

**技术指导**: 日志系统统一使用 `logging` 模块的好处：  
1. **可控级别** — ERROR/WARNING/INFO/DEBUG 可按环境过滤  
2. **自动格式化** — 时间戳、文件名、行号自动附加  
3. **多输出** — 同时写入控制台和文件，PyInstaller 环境仍可追溯  
4. **延迟求值** — `%s` 格式化在日志级别不够时跳过字符串构造，比 f-string 更高效  

### #3 `model/tool.py:17` — dataclass 可变默认值

```python
# ❌ dataclass 中 None 作为可变类型的哨兵，但 __post_init__ 只处理了 properties
@dataclass
class ToolConfig:
    properties: Dict[str, Any] = None  # 类型标注是 Dict，默认值是 None

    def __post_init__(self):
        if self.properties is None:
            self.properties = {}
```

**虽然功能正确**（`__post_init__` 修正了 None → {}），但这种写法让类型检查器（mypy/pyright）标记为错误。正确写法：

```python
from dataclasses import field
properties: Dict[str, Any] = field(default_factory=dict)
```

### #4 `base_tool.py:54` — 方法内 `import time`

```python
# ❌ 方法内导入
def finishAnnotation(self):
    import time
    data = AnnotationData(timestamp=time.time(), ...)
```

**修复**: 提到模块顶部。`time` 是标准库，不存在条件导入的需求。

**技术指导**: `import` 应遵循：  
- **模块顶部** — 无条件导入  
- **函数内** — 仅用于：重量级可选依赖、解决循环导入、惰性加载  
- 每次函数调用都执行 `import time` 浪费 0.1μs——微优化但体现编码习惯

## 审查要点及最佳实践指导

### 1. 导入规范

| 规则 | 说明 |
|------|------|
| 标准库 → 第三方 → 本地 | 分组间空行分隔 |
| 避免 `from module import *` | 命名空间污染 |
| 避免函数内 `import`（除非必要） | 可选依赖、循环导入除外 |
| 移除未使用导入 | flake8 F401 / autoflake |

### 2. 异常处理

当前代码总体良好，每个关键路径都有 try-except。但存在一个隐患：

```python
# EventBus.publish() 中
except Exception as e:
    logger.warning("[EventBus] 回调错误: %s", e)
```

回调静默吞异常是双刃剑——防止一个订阅者崩溃影响其他，但也可能隐藏 bug。建议在开发环境保留完整的 traceback：

```python
logger.warning("[EventBus] 回调错误: %s", e, exc_info=logger.isEnabledFor(logging.DEBUG))
```

### 3. 线程安全

当前已通过 `NativeEventFilter`（RegisterHotKey API）彻底解决了 `keyboard` 库的线程安全问题。审查确认：  
- 所有 Qt 对象操作在主线程  
- `QTimer.singleShot` 用于延迟操作（非用于跨线程调度）  
- `_showing_overlay` / `_opening_editor` / `_is_capturing` 重入保护到位

### 4. 代码组织建议

`controllers/editor_controller.py`（675行）承担了太多职责：
- QML 桥接控制器（Slot 装饰器）
- Pillow 渲染逻辑（`_renderAnnotationsOnImage`、`_applyMosaic`、`_drawArrow` 等）
- 导出逻辑（`saveToFile` 重复实现了 `ExportManager` 的功能）

建议后续迭代：  
- 将渲染逻辑抽到 `engines/render/` 目录  
- `ExportManager` 统一导出入口  
- `EditorController` 瘦身为纯 QML 桥接
